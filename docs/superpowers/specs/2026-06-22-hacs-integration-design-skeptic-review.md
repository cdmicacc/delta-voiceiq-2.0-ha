# Skeptic Review: HACS Integration Design Spec

## Objections

1. **Reauth trigger** — ha-best-practice
   > "**Reauth trigger:** any API call (`ToggleWater`, `Dispense`, `UsageReport`, `handWashMode`) that gets HTTP 401 raises `ConfigEntryAuthFailed`. HA automatically surfaces the standard 'Reauthenticate' affordance..."
   **Objection**: This is asserted as uniformly true across all four calls, but `ToggleWater`, `Dispense`, and `handWashMode` are service-call actions, not coordinator refreshes or `async_setup_entry` code. HA's developer docs on config-entry setup failures state `ConfigEntryAuthFailed` "must be raised from `async_setup_entry` in `__init__.py` or from the `DataUpdateCoordinator` or the exception will not be effective at triggering the reauth flow." `UsageReport` runs inside a `DataUpdateCoordinator`, so that one is fine. `ToggleWater` and `Dispense` (called from the `valve` entity's open/close handlers) and `handWashMode` (called from a service) are not coordinator refreshes — raising `ConfigEntryAuthFailed` from those code paths is not documented to trigger reauth. The spec is stating HA behavior as settled fact for code paths the docs explicitly say it doesn't cover.
   **Suggested action**: Verify the actual mechanism for surfacing reauth from a non-coordinator action (e.g. calling `entry.async_start_reauth()` directly, as docs hint at for cases outside the two documented contexts) before asserting "any API call... raises `ConfigEntryAuthFailed`" as a uniform pattern across services and the coordinator.

2. **Usage sensor unit claims** — untested-assumption / ha-best-practice
   > "`delta_voiceiq.dispense` | service | data: `amount` (number), `unit` (`ml` default, also accepts `l`/`gal`/`fl_oz`)... Usage sensors default to metric (liters) but remain user-adjustable to any HA-supported volume unit via the standard per-entity unit override... User can override the displayed unit (gal, mL, fl oz, etc.) per-entity via Settings → Entities — built into HA's water device class, no custom code."
   **Objection**: This is factually wrong, not just unverified. HA's `SensorDeviceClass.WATER` supports unit conversion only for `L, gal, m³, ft³, CCF, MCF` — confirmed directly from `developers.home-assistant.io/docs/core/entity/sensor/`. Milliliters (`mL`) and fluid ounces (`fl oz`) are **not** in that list. The spec's own service definition example uses `fl_oz` as a unit option and the sensor section claims users can display sensors in "mL, fl oz, etc." via the built-in per-entity override — neither is true for the `water` device class as documented. "built into HA's water device class, no custom code" is a confident claim that is simply false for two of the units explicitly named in the same document.
   **Suggested action**: Cut the mL/fl_oz unit-display claim for sensors entirely, or pick a device class/unit strategy that actually supports those units (e.g. don't set `device_class: water` if mL/fl_oz display matters, or accept that the per-entity override is restricted to the six units above and update the text accordingly).

3. **Token Expiry sensor entity_category** — duplicates-ha-feature / ha-best-practice (borderline)
   > "Token Expiry | `sensor` | days remaining, `entity_category: diagnostic`."
   **Objection**: HA's diagnostic-category guidance (developers.home-assistant.io) describes diagnostic entities as ones that "expose some configuration parameter or diagnostics of a device but does not allow changing it," with canonical examples being RSSI sensors, MAC address sensors, identify buttons — i.e. low-level technical telemetry about the device's connection/hardware state, generally recommended with `entity_registry_enabled_default = False` to avoid UI clutter and unnecessary recorded state changes. A "days until your auth token expires" sensor is closer to an actionable user-facing maintenance signal than hardware diagnostics, and the spec's own design *also* creates a Repair issue for the same condition once <7 days remain. Two different UI surfaces (a diagnostic sensor with full state history, and a Repair issue) for the same underlying fact is redundant, and the diagnostic categorization of a value the user is meant to act on is questionable against the stated examples.
   **Suggested action**: Decide explicitly whether the Repair issue alone is sufficient (cutting the sensor) or whether the sensor is meant for automations/dashboard display (in which case justify why it needs `entity_category: diagnostic` rather than no category, given the documented examples don't match this use case).

4. **PostAuth / UserInfo "no prior token/MAC/user-ID needed" claim** — untested-assumption
   > "The integration calls `Auth/PostAuth` with the code (no prior token/MAC/user-ID needed — verified against the existing shell script, which sends no `Authorization` header to this call), decodes the double-encoded `accessToken` and JWT `exp` claim, then calls `UserInfo` with the new token (which likewise needs only the bearer token, not MAC/user ID, since it's what *returns* those)."
   **Objection**: The `PostAuth` half is genuinely verified — `delta_token_exchange.sh` lines 61-66 confirm no `Authorization` header is sent. But the `UserInfo` half is *not* verified by the same evidence. The shell script never calls `UserInfo` at all — it only handles `PostAuth` and writes the resulting token to `secrets.yaml`. `docs/API.md` line 48-54 documents `UserInfo` as `GET /api/user/v2/UserInfo` with "Required Headers" listed at the top of the README/API doc as `Authorization: Bearer <JWT>` plus `dfc-source`/`User-Agent` — consistent with bearer-token-only auth — but nothing in the provided ground truth actually exercises a `UserInfo` call and observes its response shape. The parenthetical "since it's what *returns* those" is reasoning by assertion, not verification — it explains why it *should* work, not evidence that it *does*. The spec presents this with the same confidence as the PostAuth claim ("verified against the existing shell script") when only half of it has that backing.
   **Suggested action**: Either mark the `UserInfo` no-prior-data claim as inferred-from-API-shape rather than verified, or actually call `UserInfo` once with a bearer token (no MAC/user-ID) during planning/spike work before asserting it as a settled design fact.

5. **UserInfo device list shape / device picker** — untested-assumption
   > "**Select device** (setup only): if `UserInfo` lists more than one MAC address, show a picker; if exactly one, skip straight through. Each chosen device becomes its own config entry."
   **Objection**: Nothing in `docs/API.md` (which says only "Returns user profile, device list, MAC addresses, custom containers, and modes" with no example response body) or anywhere else in the provided ground truth shows the actual JSON shape of `UserInfo`'s device list, nor whether it labels which MAC address corresponds to which device *name* (vs. just a bare array of MACs). The spec assumes a "picker" is buildable — i.e. that there's a human-readable label per device to show the user — but this is unconfirmed. If `UserInfo` returns bare MAC addresses with no friendly name, "show a picker" silently degrades to "show a list of MAC addresses," which is a materially worse UX than implied and is a different design decision (e.g. let the user manually label devices after picking).
   **Suggested action**: Confirm the actual `UserInfo` response shape (capture one) before the plan phase; if no per-MAC display name exists, decide now whether the picker shows raw MACs, an index ("Device 1"), or something else — don't leave this as "show a picker" as if the UX is already settled.

6. **Zero-device / unexpected-shape failure mode** — inappropriate-vagueness
   > "**Select device** (setup only): if `UserInfo` lists more than one MAC address, show a picker; if exactly one, skip straight through."
   **Objection**: This only specifies behavior for "more than one" and "exactly one." It is silent on zero devices (a UserInfo call from an account with no registered VoiceIQ device, or a transient API response error that returns an empty list) and silent on a response shape the integration doesn't recognize (e.g. a future API version changing the field name/structure, given the spec's own README documents the API as "v2/v3" with version drift already occurring). This is exactly the class of unstated failure-mode gap the review brief calls out — the spec reads as if "pick a device" is the only two cases that exist, when a zero-result or malformed-result case is realistic and currently undefined.
   **Suggested action**: Add an explicit zero-devices failure branch (abort the flow with an error, e.g. `no_devices_found`) and a malformed-response branch (`cannot_connect`/`unknown` error) before this goes to planning.

7. **JWT `exp` unparseable fallback** — inappropriate-vagueness
   > "The integration calls `Auth/PostAuth` with the code... decodes the double-encoded `accessToken` and JWT `exp` claim, then calls `UserInfo` with the new token..."
   **Objection**: `delta_token_exchange.sh` has an explicit fallback path for this exact failure: if `EXP` can't be parsed from the decoded JWT payload (lines 173-180), it sets `EXP=0` and `DAYS_LEFT="unknown"` and *continues* — it does not abort the exchange, it just can't report an expiry estimate or schedule a warning. The spec's onboarding flow description never mentions this case at all; it implies `exp` decoding always succeeds ("decodes... the JWT `exp` claim" as a flat statement, no conditional). Token Expiry sensor and the 7-day Repair issue trigger both implicitly depend on having a valid `exp` timestamp. The spec doesn't say what the Token Expiry sensor or Repair issue logic do if `exp` can't be parsed on a given token (a state the original script anticipated as plausible enough to special-case).
   **Suggested action**: Specify the behavior of the Token Expiry sensor and the Repair-issue trigger when `exp` is missing/unparseable — `unknown` state, no Repair issue ever fires, or something else — as a resolved design decision, not an implicit assumption that decoding always succeeds.

8. **Config flow code validation** — inappropriate-vagueness
   > "Config flow validates the pasted code (calls `PostAuth` synchronously) and shows an inline error (`invalid_code`, `cannot_connect`) without losing the entered value."
   **Objection**: "validates the pasted code" reads as a settled mechanism, but the actual validation logic is left undefined. `PostAuth` either succeeds (302 redirect with payload) or it doesn't; the spec doesn't say what response from Delta is classified as `invalid_code` versus `cannot_connect` versus some third bucket (e.g. an expired code, a code already consumed, a malformed code string that's obviously not even attemptable). The shell script itself distinguishes at least three distinct failure points (`ERROR: No redirect from PostAuth`, `ERROR: No base64 payload in redirect URL`, `ERROR: Failed to decode base64 payload`, `ERROR: Failed to extract accessToken`, `ERROR: Token too short`) — five different failure modes collapsed in the spec into two generic error keys with no mapping specified. That's a design decision (what user-facing message corresponds to which underlying failure) being deferred while the prose reads as if the validation behavior is already decided.
   **Suggested action**: Either map the script's five failure modes to the two (or more) proposed error keys explicitly, or state plainly that this mapping is left to the plan/implementation phase — don't present "validates the pasted code" as if the validation logic is already specified.

9. **manifest.json fields / iot_class** — inappropriate-vagueness (omission)
   > "manifest.json" is listed in the architecture tree with no further description anywhere in the spec.
   **Objection**: The spec names the file but asserts nothing about its required contents, despite the rest of the doc confidently asserting other HA mechanics (config flow, coordinators, reauth) as settled. HA's required manifest fields for an integration with a config flow include `domain`, `name`, `codeowners`, `dependencies`, `documentation`, `requirements`, `config_flow: true`, and (per the docs fetched) `integration_type` for some integration types — plus `iot_class`, which while not flagged as strictly required by the fetched docs, is treated as expected/standard for any cloud-polling integration like this one (`cloud_polling` would be the fitting value given the design's poll-based coordinators). The spec is silent on `iot_class` and `config_flow: true` even though `config_flow: true` is precisely the headline feature this whole redesign is built around — it should not be implicit.
   **Suggested action**: State the manifest's required fields and `iot_class` value (`cloud_polling`) explicitly rather than leaving the file as an unlabeled bullet in a tree diagram.

10. **Translations / strings.json scope** — inappropriate-vagueness (omission)
    > "strings.json" and "translations/en.json" appear in the architecture tree with no description of what they must contain.
    **Objection**: The spec describes a multi-step config flow (provider picker, instructions step, code-paste step, device picker, reauth-confirm step) and multiple new error/abort conditions (`invalid_code`, `cannot_connect`, plus whatever covers objections 6 and 8 above) — every one of those needs a corresponding `strings.json` entry under the `config` key (`step`, `error`, `abort` sections per HA's translation docs) for the UI to render anything other than raw keys. The spec never states this requirement or accounts for the volume of UI text this design implies (provider selection copy, DevTools instructions copy, paste-field copy, per-step titles). This isn't a paperwork detail — without it the config flow described literally cannot render in the HA UI. Listing the file in a tree without acknowledging what populating it requires understates the actual scope of "onboarding & reauth flow."
    **Suggested action**: Either enumerate the strings.json keys needed for the steps/errors already described, or explicitly flag this as plan-phase work — don't let the architecture diagram imply it's a thin pass-through file.

11. **services.yaml dispense target wording** — inappropriate-vagueness
    > "`delta_voiceiq.dispense` | service | data: `amount` (number), `unit` (`ml` default, also accepts `l`/`gal`/`fl_oz`), targets the device/valve entity. Converts to ml and calls `Dispense`."
    **Objection**: "targets the device/valve entity" elides a real decision. HA's services.yaml conventions (per the docs fetched) distinguish `entity_id` targets (service "operates on or requires a specific entity") from `device_id` targets (service "operates on a device as a whole") — these are different `target:` selector configurations with different implications for how the service is exposed in the UI/automations editor (entity picker vs. device picker) and how the handler resolves which `DeltaVoiceIQClient`/MAC to call against. "device/valve entity" reads as if these are interchangeable, but the spec needs to pick one. Given the design's stated principle of "no loose helpers... entities and services tied to it" (the device), and the explicit guidance to "target the thing the action actually acts on" — is the action's target conceptually the device or the valve entity? This is unresolved.
    **Suggested action**: Pick `device_id` or `entity_id` as the explicit service target type and update services.yaml's described schema accordingly — don't present both as a slash-option in a one-line table cell.

12. **Multi-coordinator-per-interval architecture** — ha-best-practice (flag, not necessarily wrong)
    > "Four `DataUpdateCoordinator` instances per entry, one per `UsageReport` interval, preserving today's cadence... | Coordinator | Interval | Poll cadence | Today | 0 | 10 min | Week | 1 | 1 hour | Month | 2 | 5 hours | Year | 3 | 24 hours |"
    **Objection**: HA's coordinator design guidance (fetched from developers.home-assistant.io) frames `DataUpdateCoordinator` around the idea of "a single periodical poll on this endpoint" feeding multiple entities from one shared data structure — the canonical pattern is one coordinator per logical data source, not one per derived cadence. Here the four "sources" are actually the same `UsageReport` endpoint called with four different `interval` query-param values, which could be framed as one coordinator with four pieces of data updated on different internal timers, or four independent coordinators (since they really do have independent legitimate cadences and failure domains). The spec asserts the four-coordinator design as obviously correct ("preserving today's cadence") without acknowledging this is a real architectural choice against the documented single-coordinator-per-source pattern, and doesn't explain why splitting wins here (e.g. independent backoff/availability per interval) versus the documented norm.
    **Suggested action**: This may well be the right call given four genuinely different poll cadences against the same endpoint — but the spec should say so explicitly as a deliberate departure from the "one coordinator, one source" default, with the reasoning (independent staleness/availability per interval) stated, rather than presenting it as the unremarkable default.

13. **README migration notes — entity ID continuity claim** — untested-assumption
    > "Any automations/scripts referencing the old `input_boolean.delta_faucet_state` / `sensor.delta_faucet_usage_*` / `script.delta_faucet_*` entities need updating to the new `valve.*` / `sensor.*` entity IDs and the new services."
    **Objection**: This is fine as a migration-notes bullet, but note it silently confirms there is no migration *path* (e.g. no attempt to preserve old entity_ids via `unique_id`/entity registry tricks, no `unique_id`-based smooth cutover) — every existing user's automations, scripts, and dashboard break and must be hand-edited. That may be an acceptable and even unavoidable consequence of the redesign, but the spec states it as a flat fact ("need updating") rather than acknowledging it as a cost of the design being signed off on. Given the Non-goals section explicitly defers the dashboard card to "a separate future effort," but the entity-ID break extends well beyond the dashboard to every automation/script a user already wrote — that blast radius deserves to be named as a tradeoff, not buried as a migration-notes afterthought.
    **Suggested action**: No code change needed, but name this explicitly as an accepted breaking-change cost in the Goals/Non-goals section, not only in the migration footnote.

## Summary

13 objections raised, spanning all four requested axes, with the most severe concentrated in untested-assumptions (5) and inappropriate-vagueness (6), plus 1 best-practice violation that is factually wrong (not just unverified) and 1 borderline duplication/over-use of a diagnostic category. The single most serious objection is **#2**: the spec asserts as a settled, confidently-stated fact ("built into HA's water device class, no custom code") that users can display usage sensors and the dispense service's unit option in milliliters or fluid ounces via HA's standard per-entity unit override — but HA's `water` device class only supports L, gal, m³, ft³, CCF, and MCF. This isn't a documentation-checking gap, it's a wrong claim embedded in the design's stated Goals ("Usage sensors default to metric (liters) but remain user-adjustable to any HA-supported volume unit") and repeated in the entity table — it should be corrected before planning, since it affects both the sensor design and the dispense service's `unit` field defaults. The reauth-from-service-calls claim (#1) is the second most serious: it asserts a specific HA mechanism (`ConfigEntryAuthFailed` triggering reauth from any of four API call sites) that the framework's own docs say is only effective from coordinator/setup contexts, meaning three of the four named call sites may not actually trigger reauth the way the spec describes. Both should be resolved — by checking the actual HA source/behavior, not by more confident restating — before this spec is treated as implementation-ready.
